# v0.1.23

This hotfix closes an old-database migration gap found during the real v0.1.22 upgrade rehearsal. It also makes backup recovery migration-aware and keeps large model caches on the configured D-drive runtime path.

## Tenant-safe audit-log migration

- Legacy `recall_log`, `like_recall_log`, `uncollect_log`, and `unlike_log` tables are rebuilt with composite `(user_id, item_id)` foreign keys whenever their definitions do not exactly match the current schema.
- The four-table migration runs in one immediate transaction and validates source and migrated row counts, composite-key orphans, foreign keys, indexes, and `sqlite_sequence` before commit.
- Explicit audit IDs and every existing business field are retained. A failed migration rolls all four tables back and leaves no migration table behind.
- Legacy `job_queue` tables missing the user foreign key are rebuilt transactionally while preserving rows, IDs, retry state, and `sqlite_sequence`; rows for unknown users are rejected before cutover.
- Older databases that do not yet have tenant columns can now reach column migration before tenant indexes are created.
- User removal deletes dependent audit rows and vector/FTS partitions before parent content, then clears reindex markers, jobs, and sessions.

## Migration-aware backup and restore

- Database doctor now reports both SQLite integrity and foreign-key health.
- Backup validation distinguishes a known, safely migratable legacy schema from real data corruption by migrating an isolated SQLite snapshot and rechecking row counts, sequences, integrity, and foreign keys.
- Backups containing real orphan rows remain blocked before the active database or worker connections are touched.
- Validation and restore use SQLite's backup API, so committed WAL frames are included instead of silently reading or copying only the stale main file.
- Restore prepares and migrates a self-contained temporary snapshot, validates it, checkpoints the current target, and uses same-volume atomic replacement. Replacement failures leave the previous target readable and preserve a safety backup.
- Restore blocks new in-process database connections during cutover, refuses to continue if the background worker cannot stop, and distinguishes a committed restore from a later runtime-restart failure.
- Target WAL/SHM/journal files are quarantined before the main-file replacement. Missing targets with orphan hot journals cannot replay old pages into the restored database, and cleanup failures are reported without turning a committed restore into a false failure.
- Invalid or empty targets are retained as `.corrupt` forensic copies instead of being advertised as protected, recoverable SQLite backups.
- Delivery manifests require SHA256 plus `users`/`favorites`/`likes` counts, reject non-empty WAL or rollback-journal sidecars, and recheck the locked source immediately before copying to close validation-to-restore races.

## Windows runtime cache and upgrade gate

- The installed launcher sets `HF_HOME` and `SENTENCE_TRANSFORMERS_HOME` below `D:\codexDownload\douyinclaude-runtime`, preventing the embedding model from defaulting to the user-profile cache.
- The in-place upgrade gate now uses the exact public v0.1.22 installer as its old binary and verifies v0.1.23 over a populated legacy audit-log and search schema.
- Upgrade acceptance covers installer-created backup preservation, composite foreign keys, audit IDs and sequences, automatic 收藏/喜欢 reindex recovery, search hits, idempotent second initialization, and uninstall data retention.

## Install notes

- The installer is still unsigned. Windows SmartScreen may warn on first download. 当前安装包仍然未签名。
- First launch can download Python dependencies, Playwright browser assets, and local model files. 首次启动可能需要较长时间。
- Runtime downloads and caches remain under `D:\codexDownload\douyinclaude-runtime`, with `UV_LINK_MODE=copy` for cross-drive installs.
- The launcher still runs 启动前健康检查 before preparing dependencies.
- Normal startup no longer opens that preparation page during normal startup; status files and logs remain available for troubleshooting.
- Installer Finish and the main shortcuts run the launcher hidden in the background.
- The launcher waits until the local Web endpoint is actually reachable before opening the browser.
- The installer always shows the installation directory page during fresh install, reinstall, and upgrade.
- The Start Menu 控制入口 remains `Douyin Recall Control`, and it prints a 状态摘要 before the menu.
- Existing 健康检查 and repair behavior remain available through `Douyin Recall Health Check` and `Douyin Recall Repair State`.
- Use `Douyin Recall Prepare Runtime` when dependency, browser, or database preparation fails; it does not start the local web service.
- Local operations remain available through `Douyin Recall Stop Service` and `uv run python -m src.cli stop`.
- Backup and recovery entries remain available: `Douyin Recall Backup Now`, `Douyin Recall Backups`, `Douyin Recall Restore Center`, `Douyin Recall Verify Backup`, and `Douyin Recall Rollback Check`.
- Use `Douyin Recall Verify Backup` or `recall verify-backup` for a read-only check of the latest backup in `data\exports`.
- Use `rollback-from-manifest` for a read-only delivery rollback check before explicitly applying any restore.
- Before `rollback-from-manifest --apply` targets the active database, stop the local Web service with `recall stop`; the CLI refuses a live-service restore.
- Installer upgrade safety backups use the `pre-install-recall-*.db` filename pattern.
- Use `Douyin Recall Account Recovery` or `/auth` to rebind an account; maintenance remains available at `/maintenance`.

After all release gates pass, download `DouyinRecallSetup.exe` from the GitHub Release and install it over v0.1.22. The installer creates a pre-install SQLite backup before replacing application files.
