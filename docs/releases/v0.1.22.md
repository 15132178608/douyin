# v0.1.22

This release closes the remaining upgrade-search, Web security, route ownership, transactional item-action, and Windows delivery gaps identified after v0.1.21.

## Upgrade safety and automatic search recovery

- Upgrading a legacy vec/FTS schema records durable recovery work for every user and content kind that still has active 收藏 or 喜欢 data.
- The schema change and its per-user recovery markers are committed in the same SQLite transaction, so an interrupted migration cannot silently lose the need to rebuild search.
- Worker startup materializes pending markers as idempotent `force=true` index jobs; QR authorization and manual standard maintenance are no longer prerequisites for restoring search.
- Force indexing replaces only the affected user's partition and preserves the previous partition when encoding fails.
- Recovery remains retryable beyond the normal job-attempt budget, and a marker is completed only when the active item IDs exactly match both vector and FTS IDs.

## Web security and account isolation

- Session-user database resolution now runs outside the async event loop.
- Invite claiming creates the user and session and consumes the invite in one SQLite transaction; concurrent single-use claims cannot both succeed.
- Failed login attempts are persistently rate-limited by direct client IP and invite fingerprint, and authenticated non-loopback binds require secure session cookies.
- The effective CLI `--host` override is validated before database initialization, server-state writes, or Uvicorn startup, and the same effective host is passed to application-lifespan validation.
- Login redirects accept only safe local absolute paths, including after repeated decoding, so encoded slash and backslash forms cannot escape to an external site.
- User browser profiles honor the configured data root while using collision-resistant directories and narrowly scoped legacy-profile reuse.
- Avatar downloads require configured HTTPS hosts, validate every redirect and resolved address, reject mixed private/public DNS answers, enforce image type and size limits, and pin each TCP connection to the validated address while preserving Host, SNI, and certificate checks.

## Real route ownership

- The former monolithic `src/web/routes/content.py` has been removed.
- Browse, search, timeline, and library views now live in `browse`; category behavior lives in `categories`; selected-item export and 收藏/喜欢 mutations live in `item_actions`.
- Authentication, setup, jobs, maintenance, and media routes own their implementations instead of rebinding functions from a shared content module.
- Shared behavior has moved into route-free services such as `content_items`, `content_state`, `douyin_auth`, `job_service`, and `item_action_service`.
- `src/web/app.py` remains application assembly only; there is no wildcard import, dynamic `__getattr__` compatibility shim, or route-to-route implementation import.

## Atomic 收藏/喜欢 actions

- Single and batch uncollect/unlike requests mark local items removed, append the correct 收藏 or 喜欢 audit row, and enqueue remote work through the same explicit SQLite connection and `BEGIN IMMEDIATE` transaction.
- Any failure rolls back the entire unit, including jobs already inserted earlier in a batch.
- Repeated and concurrent requests are idempotent: an item is queued once, already-removed items are reported without duplicate jobs, and different users and content kinds stay isolated.
- Track/open updates `last_recalled_at` and writes the matching recall log atomically; missing or foreign items return 404 without an orphan log.
- Existing paths and response behavior remain compatible with the Web UI.

## Windows PR CI and release verification

- Pull requests to `main` now run `uv run --locked pytest -q` with Python 3.11 on `windows-latest`, the supported delivery platform.
- The Windows installer workflow also builds on pull requests, uses read-only repository permissions for untrusted build work, and grants release-write permission only to the tag-only draft-publishing job.
- GitHub Actions are pinned to reviewed revisions, checkout credentials are not persisted, and the installer artifact must exist before upload or draft release creation.
- Installed-build test fixtures declare and isolate their Web profile prerequisites, while custom Release Gate evidence directories are honored consistently.
- A green pull request is required but does not replace fresh-install, in-place-upgrade, backup/restore, installed-smoke, and installer-hash verification before publishing v0.1.22.

## Verification required before publishing

- The full locked pytest suite and required Windows PR checks must pass.
- Test a fresh install and an in-place upgrade from an older schema without QR login or manual maintenance; after the background worker finishes, existing 收藏 and 喜欢 searches must return their original items.
- Verify concurrent 收藏/喜欢 removal retries, rollback after injected failure, and missing/foreign track-open requests against a file-backed SQLite database.
- Run installed-build smoke, database safety, backup/restore, Web and SQL performance, acceptance evidence, and manifest rollback dry-run checks through Release Gate.
- Record the final `DouyinRecallSetup.exe` SHA256 in the installer-enabled Release Gate evidence.

## Install notes

- The installer is still unsigned. Windows SmartScreen may warn on first download. 当前安装包仍然未签名。
- First launch can download Python dependencies, Playwright browser assets, and local model files. 首次启动可能需要较长时间。
- Runtime downloads and caches remain under `D:\codexDownload\douyinclaude-runtime`, with `UV_LINK_MODE=copy` for cross-drive installs.
- The launcher still runs 启动前健康检查 before preparing dependencies.
- Normal startup no longer opens that preparation page during normal startup; status files and logs remain available for troubleshooting.
- Installer Finish and the main shortcuts run the launcher hidden in the background.
- The launcher waits until the local Web endpoint is actually reachable before opening the browser.
- The installer always shows the installation directory page during fresh install, reinstall, and upgrade.
- The Start Menu 控制入口 remains `Douyin Recall Control`, and it prints a 状态摘要 before the menu.
- Existing 健康检查 and repair behavior remain available through `Douyin Recall Health Check` and `Douyin Recall Repair State`.
- Use `Douyin Recall Prepare Runtime` when dependency, browser, or database preparation fails; it does not start the local web service.
- Local operations remain available through `Douyin Recall Stop Service` and `uv run python -m src.cli stop`.
- Backup and recovery entries remain available: `Douyin Recall Backup Now`, `Douyin Recall Backups`, `Douyin Recall Restore Center`, `Douyin Recall Verify Backup`, and `Douyin Recall Rollback Check`.
- Use `Douyin Recall Verify Backup` or `recall verify-backup` for a read-only check of the latest backup in `data\exports`.
- Use `rollback-from-manifest` for a read-only delivery rollback check before explicitly applying any restore.
- Installer upgrade safety backups use the `pre-install-recall-*.db` filename pattern.
- Use `Douyin Recall Account Recovery` or `/auth` to rebind an account; maintenance remains available at `/maintenance`.

After all release gates pass, download `DouyinRecallSetup.exe` from the GitHub Release and install it over the previous version to upgrade. The installer creates a pre-install SQLite backup before replacing application files.
