# v0.1.25

This release turns Windows runtime preparation and first startup into visible, retryable flows, while tightening the exact-binary release gate introduced after v0.1.24. It does not remove or rewrite user content.

## Guided Windows runtime preparation

- Fresh interactive installs offer a default-selected task that prepares uv, production dependencies resolved from the bundled `uv.lock`, the exact Playwright browser assets, the database, and final runtime status.
- The installer displays five real stages plus the latest trustworthy tool activity; it does not invent a global byte percentage when the underlying tool cannot provide one.
- A failed preparation can be retried immediately. Choosing cancel defers the work, still completes installation, and suppresses the immediate hidden application launch.
- Users can clear the preparation task before installation. Silent installs and in-place upgrades skip this fresh-only online step and remain suitable for offline or managed deployment.
- `Douyin Recall Prepare Runtime` resumes unfinished work later, skips stages that pass strict readiness checks, and refreshes the prepared fingerprint only after all postconditions succeed.

## Exact readiness and actionable startup

- Playwright readiness is derived from the installed manifest and requires the current Chromium, headless shell, FFmpeg, and Winldd revisions, their completion markers, and their expected executables.
- A successful browser command with incomplete postconditions is repaired once with `--force`; persistent incompleteness fails explicitly instead of writing a false prepared marker.
- The installer, Start Menu recovery entry, and main launcher share preparation status, logs, lock ownership, and the fingerprint marker; status JSON/pages and the fingerprint marker are updated atomically.
- A cross-process preparation lock permits one owner. The helper Job Object keeps uv and Playwright children within the owner's lifecycle so a cancelled or crashed wrapper cannot leave tools modifying the runtime after the lock is released.
- Unprepared startup uses one seven-stage `startup-status.html` page with elapsed time and current activity. The same page redirects after the Web endpoint is reachable and retains the failed stage, likely cause, error summary, and next action when startup fails.
- Long and Unicode installation paths are passed through dedicated runner/worker scripts instead of fragile inline command quoting.

## Exact-binary release evidence

- Release Gate accepts an existing installer only when its absolute path, non-zero size, ProductVersion, SHA256, and Authenticode state are recorded and validated.
- Final Release Check requires fresh machine-readable Release Gate artifacts from the current invocation and fails closed on missing, stale, failed, or stringly-typed evidence.
- Tag builds still create a Draft Release. The exact workflow-built asset must be downloaded and retested because a locally compiled EXE cannot substitute for the Draft asset.
- The build script rejects disagreement between `pyproject.toml` and the Inno version and rejects a GitHub version tag that does not equal `v<project-version>`.

## Verification required before publishing

- The full locked pytest suite and required Windows pull-request checks must pass.
- Run isolated GUI cases for default-selected preparation, task deselection, failed-then-Retry, failed-then-Defer, and verify their logs, state files, markers, locks, child processes, and launch behavior.
- Run an offline silent fresh install and verify it does not force runtime downloads or start the browser or Web service.
- Run the exact public-v0.1.24-to-v0.1.25 in-place-upgrade gate and confirm the installer-created pre-upgrade backup, existing database, automatic 收藏/喜欢 reindex recovery, search hits, and uninstall data retention.
- Run Final Release Check against the candidate installer and record its ProductVersion, size, SHA256, and Authenticode state.
- After tagging, download the exact Draft Release asset and repeat installed-build QA, the v0.1.24 upgrade gate, GUI-critical checks, and final release checks before making the Release public.

## Install notes

- The installer is still unsigned. Windows SmartScreen may warn on first download. 当前安装包仍然未签名。
- GUI 全新安装默认准备 Python dependencies and Playwright browser assets inside the installer; 首次启动 and first indexing can still take time because the local embedding model is downloaded by the first index job, not by the installer.
- Runtime downloads and caches remain under `D:\codexDownload\douyinclaude-runtime`, with `UV_LINK_MODE=copy` for cross-drive installs.
- The launcher still runs 启动前健康检查 before preparing dependencies.
- An already-prepared normal start no longer opens that preparation page during normal startup; status files and logs remain available for troubleshooting.
- Installer Finish and the main shortcuts run the launcher hidden in the background.
- The launcher waits until the local Web endpoint is actually reachable before redirecting the startup page.
- The installer always shows the installation directory page during fresh install, reinstall, and upgrade.
- The Start Menu 控制入口 remains `Douyin Recall Control`, and it prints a 状态摘要 before the menu.
- Existing 健康检查 and repair behavior remain available through `Douyin Recall Health Check` and `Douyin Recall Repair State`.
- Use `Douyin Recall Prepare Runtime` when dependency, browser, or database preparation fails; it does not start the local web service.
- Local operations remain available through `Douyin Recall Stop Service` and `uv run python -m src.cli stop`.
- Backup and recovery entries remain available: `Douyin Recall Backup Now`, `Douyin Recall Backups`, `Douyin Recall Restore Center`, `Douyin Recall Verify Backup`, and `Douyin Recall Rollback Check`.
- Use `Douyin Recall Verify Backup` or `recall verify-backup` for a read-only check of the latest backup in `data\exports`.
- Use `rollback-from-manifest` for a read-only delivery rollback check before explicitly applying any restore.
- Before `rollback-from-manifest --apply` targets the active database, stop the local Web service with `recall stop`; the CLI refuses a live-service restore, an active configured listener, or an acquired database runtime lock.
- Installer upgrade safety backups use the `pre-install-recall-*.db` filename pattern.
- Use `Douyin Recall Account Recovery` or `/auth` to rebind an account; maintenance remains available at `/maintenance`.

After all release gates pass, download `DouyinRecallSetup.exe` from the Draft GitHub Release and install it over v0.1.24. The installer creates a pre-install SQLite backup before replacing application files.
