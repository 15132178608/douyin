# 首次启动向导设计

## 背景

当前项目已经可以通过 GitHub Release 提供 `DouyinRecallSetup.exe`，安装脚本会准备 `.env`、依赖、Playwright Chromium、数据库并启动本地 Web 服务。但普通用户第一次打开后仍需要理解“绑定账号、同步、索引、查看后台队列”这些工程概念，容易卡在第一步。

阶段 0.1 的目标是把“能安装”推进到“普通人能完成第一次使用”。当前产品定位仍是个人本地工具，默认只监听 `127.0.0.1`，不引入多人云服务、桌面壳、自动更新或安装包签名。

## 目标

让首次用户打开浏览器后进入一个清晰的 `/setup` 流程，按顺序完成：

1. 确认本地环境已准备好。
2. 扫码绑定抖音账号。
3. 选择同步收藏、喜欢，或两者都同步。
4. 在同一页面看到后台任务进度。
5. 数据可用后进入首页。

成功标准：

- 新用户不需要阅读 README 也能知道下一步该点哪里。
- 老用户已有数据时不会被强制打断。
- 授权、同步、索引仍复用现有稳定链路。
- 出错时显示中文原因和重试入口。

## 非目标

本阶段不做以下内容：

- Tauri / Electron 桌面壳。
- 自动更新。
- 安装包签名。
- 多人云服务或公网部署体验。
- SMTP 邮件配置向导。
- 自动分类全流程。首次完成页只给出后续整理入口，避免第一次使用时被模型下载和聚类耗时拖住。

## 推荐方案

采用 Web 首启向导，不改安装包主架构。

安装脚本继续负责把本地服务启动起来并打开浏览器。Web 端新增 `/setup` 页面和轻量状态判断模块。用户是否需要进入向导由本地数据库状态决定：

- 没有收藏和喜欢数据：显示首页空态，并突出“开始设置”入口。
- 已有数据：正常进入首页，不自动跳转 `/setup`。
- 用户主动访问 `/setup`：始终可以重新走绑定、同步和索引步骤。

这样可以避免“老用户打开首页被流程打断”，同时给首次用户明确路径。

## 架构

新增 `src/onboarding.py`，负责汇总首启状态。它只读取本地状态，不启动浏览器、不执行抓取、不写复杂业务逻辑。

主要职责：

- 检查数据库路径是否存在。
- 统计当前用户收藏和喜欢数量。
- 统计收藏和喜欢索引数量。
- 判断是否存在 Playwright profile 目录。
- 读取当前用户已保存的抖音昵称、头像等展示信息。
- 汇总最近后台任务状态。
- 给出 `needs_setup`、`has_any_items`、`has_profile` 等布尔字段。

`src/web/app.py` 新增 `/setup` 路由，并复用现有接口：

- `/auth/start`：启动扫码绑定。
- `/auth/status`：轮询二维码和授权结果。
- `/auth/qr-image`：展示二维码。
- `/jobs/sync`：加入同步收藏/喜欢任务。
- `/jobs/index`：加入索引任务。
- `/jobs/status`：展示后台队列。

新增模板：

- `src/web/templates/setup.html`：完整首启向导页面。
- `src/web/templates/_setup_status.html`：用于 HTMX 轮询刷新状态概览。

首页模板在空数据时增加明显入口，链接到 `/setup`。

## 页面流程

`/setup` 页面使用单页分区，不做复杂状态机跳转。

区域 1：本地环境

- 显示数据库、浏览器资料目录、后台队列状态。
- 正常项显示“已准备好”。
- 异常项显示中文提示，例如“数据库还没有初始化，请重新启动应用”。

区域 2：绑定账号

- 嵌入现有账号绑定状态组件。
- 未绑定时显示“开始扫码绑定”。
- 二维码出现后显示扫码说明。
- 成功后显示已绑定账号昵称。

区域 3：同步数据

- 提供三个按钮：
  - 同步收藏
  - 同步喜欢
  - 同步收藏和喜欢
- 点击后调用现有 `/jobs/sync`。
- 页面提示“任务已加入后台队列，可以等待进度变化”。

区域 4：生成索引

- 当收藏或喜欢数量大于 0 时显示索引按钮。
- 提供两个按钮：
  - 索引收藏
  - 索引喜欢
- 点击后调用现有 `/jobs/index`。
- 不强制用户等待索引完成才能进入首页；但搜索体验提示需要索引后更完整。

区域 5：完成

- 当收藏或喜欢数量大于 0 时显示“进入首页”。
- 如果已索引数量大于 0，文案提示“可以开始搜索”。
- 如果还没索引，文案提示“可以先浏览，稍后回来生成搜索索引”。

## 数据流

页面初次加载：

1. Web 路由调用 `onboarding.get_onboarding_status(user_id)`。
2. 模板根据状态渲染各区块。
3. `_setup_status.html` 每 5 秒轮询，更新数量、索引、队列状态和下一步提示。

用户点击绑定：

1. 前端 HTMX 调用 `/auth/start`。
2. 现有后台线程启动 `DouyinCrawler.authorize_by_qr`。
3. 页面继续轮询 `/auth/status` 获取二维码和结果。

用户点击同步或索引：

1. 前端 HTMX 调用 `/jobs/sync` 或 `/jobs/index`。
2. 现有 `jobs` worker 执行任务。
3. `/setup` 轮询 `_setup_status.html` 和 `/jobs/status` 显示进度。

## 错误处理

错误展示原则：

- 用户看到中文解释，不暴露 Python traceback。
- 任务失败时给出“查看后台队列”和“重试同步/索引”的入口。
- 授权失败时保留现有重新绑定按钮。
- 数据库不可用时提示重新启动应用，不自动尝试破坏性修复。

具体映射：

- `profile_exists=false` 且未绑定：提示“需要先扫码绑定”。
- 最近同步任务 failed：显示“同步失败，请检查登录态是否过期，然后重试”。
- 最近索引任务 failed：显示“索引失败，可能是模型下载或网络问题，请稍后重试”。
- 没有任何数据：显示“还没有同步收藏或喜欢”。

## 测试

新增和更新测试：

- `tests/test_onboarding.py`
  - 空数据库状态下 `needs_setup=true`。
  - 插入收藏或喜欢后 `has_any_items=true`。
  - 插入向量索引后 indexed 计数正确。
  - profile 目录存在时 `has_profile=true`。

- `tests/test_web_templates.py`
  - `setup.html` 包含环境、绑定、同步、索引、完成五个区块。
  - 首页空态包含 `/setup` 入口。
  - setup 页面复用 `/auth/start`、`/jobs/sync`、`/jobs/index`、`/jobs/status`。

- 现有测试保持通过：
  - `tests/test_jobs.py`
  - `tests/test_web_templates.py`
  - `tests/test_parser.py`
  - `tests/test_windows_packaging.py`

## 发布影响

本阶段不改变安装包结构。完成后发布新的 `v0.1.x` 标签即可让 Release 自动生成新版 `DouyinRecallSetup.exe`。

README 需要更新：

- Windows 安装包下载后，第一次启动会进入 `/setup`。
- 首次同步和索引可能需要较长时间。
- 数据仍保存在本地 `data/`，卸载程序不会自动上传或共享数据。
